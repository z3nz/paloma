**Files:** `server.js`, `package.json`, `README.md`, `data/`, `node_modules/`  
**Data dir is empty** — `~/paloma-memory/data/` has no JSON files.

### Config File: `~/.paloma/mcp-settings.json`

- Lives **outside the repo** (user home dir, not version-controlled)
- Generated by `scripts/setup-mcp.sh` which runs on `npm install` (via `postinstall`)
- Preserved API keys on regeneration (grep-based extraction, no jq required)
- Contains absolute paths — tightly coupled to the machine that ran setup

### How the Bridge Loads Config

1. `bridge/config.js` reads `~/.paloma/mcp-settings.json` at startup
2. Parses `.servers` object → passes each entry to `McpManager.startAll()`
3. `McpManager` spawns each server via `StdioClientTransport` (child process, stdio)
4. Each server announces its tools via MCP `tools/list` → stored in memory
5. `McpProxyServer` exposes all tools with `serverName__toolName` prefix to Claude CLI

## The Memory Server Situation

This is the most important finding. **There are two nearly-identical memory servers:**

### External Repo (`~/paloma-memory/server.js`)
- What `mcp-settings.json` actually points to
- Data storage: `./data/` relative to server.js → `~/paloma-memory/data/`
- **Data directory is empty** — no memories stored there
- Server name in MCP: `paloma-memory`
- Has its own `package.json`, `node_modules/`, separate git history

### In-Repo (`mcp-servers/memory.js`)  
- Not currently used by mcp-settings.json (orphaned!)
- Data storage: `~/.paloma/memory/{collection}.json`
- Server name in MCP: `memory`

### Key Differences

| Aspect | External (`~/paloma-memory/`) | In-Repo (`mcp-servers/memory.js`) |
|--------|------------------------------|----------------------------------|
| Data path | `./data/` (sibling to server.js) | `~/.paloma/memory/` |
| MCP server name | `paloma-memory` | `memory` |
| Stats includes dataDir | YES | NO |
| Currently used | YES (in mcp-settings.json) | NO |

### Critical Issue

The external repo stores data in `~/paloma-memory/data/` which is **empty**. This means either:
1. The memory server has never been used (unlikely — we've used `memory_store` in sessions)
2. OR Claude's MCP tools are routing to a different memory server than expected

Looking at the current `mcp-settings.json`: the `memory` server is `~/paloma-memory/server.js`. But the in-repo `mcp-servers/memory.js` is not in the config at all. So any memories stored during Claude Code sessions (like through `mcp__paloma__memory__*`) are going to `~/paloma-memory/data/` which shows as empty.

Actually — reviewing again — the Paloma *bridge* launches memory from `~/paloma-memory/server.js`. But Claude Code (this CLI session) connects via the MCP proxy on port 19192, which routes through the bridge. So memories stored via Claude Code sessions in Paloma go to `~/paloma-memory/data/` (empty). The Claude Code MEMORY.md system (file-based, at `~/.claude/projects/.../memory/`) is separate and unrelated.

**The in-repo `mcp-servers/memory.js` is dead code in the current setup.**

## How `mcp-settings.json` Is Generated and Consumed

### Generation (setup-mcp.sh)
1. Checks for existing file and extracts API keys (grep-based, no jq)
2. Optionally clones `paloma-memory` repo (SSH), falls back to copying in-repo memory.js
3. Detects platform (`uname -s`)
4. Creates Python venv for voice if Python 3 is available
5. Writes `mcp-settings.json` as a heredoc with:
   - Absolute `$PALOMA_DIR` paths for in-repo servers
   - `npx -y` for external packages
   - Injected API keys
6. Creates `.paloma/mcp.json` (permissions/autoExecute) if missing

### Consumption (bridge/config.js + McpManager)
- `bridge/config.js` reads the file at bridge startup
- Returns `config.servers` object
- `McpManager.startAll()` spawns each as a stdio subprocess
- No hot-reload — changes require bridge restart

### The Path Coupling Problem
The config embeds absolute paths like `/Users/adam/Projects/paloma/mcp-servers/web.js`. This breaks if:
- The project is moved to a different directory
- Someone else clones the repo to a different path
- The setup script is re-run, it regenerates correctly — but only if run from the right directory

## Recommendations for Chart

### 1. Install All 5 External Packages as npm Dependencies

Add to `package.json` `dependencies` (not devDependencies — they run at runtime):
```json
"@modelcontextprotocol/server-brave-search": "latest",
"@mseep/git-mcp-server": "latest",
"@kevinwatt/shell-mcp": "latest",
"@modelcontextprotocol/server-filesystem": "latest",
"@thelord/mcp-cloudflare": "latest"
```

After install, `node_modules/.bin/` has executables. `setup-mcp.sh` generates mcp-settings.json using:
```json
"command": "node",
"args": ["$PALOMA_DIR/node_modules/@pkg/dist/index.js"]
```
instead of `npx -y`. This pins versions and removes the network dependency at startup.

**Challenge:** Need to know each package's actual entry point (dist/index.js path). Will need to check post-install. Alternatively use `node_modules/.bin/{binary-name}` which is simpler.

### 2. Consolidate Memory: Drop External Repo, Use In-Repo

**Action:** Point `memory` in mcp-settings.json to `mcp-servers/memory.js` (already in-repo).

The key decision is where to store memory data:
- **Keep `~/.paloma/memory/`** (current in-repo default) — user-local, not in git, correct behavior
- Remove the `paloma-memory` external repo dependency entirely
- Update `setup-mcp.sh` to not clone the external repo
- The in-repo `mcp-servers/memory.js` is already feature-equivalent to the external repo's `server.js`

**Data migration note:** `~/paloma-memory/data/` is empty, so no migration needed.

### 3. Add a Template for mcp-settings.json

Commit `scripts/mcp-settings.template.json` to the repo with `$PALOMA_DIR` and `$HOME` placeholders. `setup-mcp.sh` replaces placeholders rather than embedding the full JSON as a heredoc. This makes adding new servers a code change (tracked in git) rather than a script change buried in a heredoc.

### 4. Keep API Keys Outside the Repo

`BRAVE_API_KEY`, `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ZONE_ID` must stay in `~/.paloma/mcp-settings.json` — never committed. The template approach handles this: template has placeholder values, setup script injects real keys from existing config or prompts.

### 5. The Filesystem Server's Scope Argument

The `@modelcontextprotocol/server-filesystem` takes the allowed directory as a CLI arg: `node dist/index.js /Users/adam`. This needs to be `$HOME` dynamically in the config — setup-mcp.sh already handles this. On macOS `/Users/adam`, on Linux `/home/adam`.

### 6. Voice — No Change Needed

The voice server is already fully in-repo and works via `kokoro_env/` which setup-mcp.sh creates. The only platform concern is that `kokoro_env/` is macOS/Linux only — no Windows native support. This is acceptable for now (WSL works).

## Open Questions for Chart

1. **Entry points:** What's the actual binary/entry path for each npm-installed package? Need to check `node_modules/.bin/` after install, or inspect each package's `bin` field in their package.json. This determines the exact `command`/`args` in mcp-settings.json.

2. **Version pinning:** Use `"latest"` or pin specific versions? Latest is simpler to maintain but could break on updates. Specific versions (e.g., `"@mseep/git-mcp-server": "2.1.4"`) are safer. Recommend starting with specific versions, updating intentionally.

3. **The `@kevinwatt/shell-mcp` tool:** It's a third-party package with less active maintenance. Is it worth replacing with our own in-repo shell server (like `exec.js` but read-only)? We'd have full control. Medium complexity but high reliability.

4. **Memory data path:** Should we migrate the memory data dir to be more explicit? Currently `~/.paloma/memory/` — this is user-local and correct. The external repo's `./data/` approach (alongside the server file) is actually worse for user data.

5. **mcp.json permissions:** The `.paloma/mcp.json` autoExecute list uses server names. If server names change, permissions break. The restructure should keep server names identical to avoid breaking autoExecute configs.

## Summary for Chart

The restructure is **achievable and low-risk**. All 5 external packages are npm-installable. The memory situation resolves cleanly by using the in-repo `mcp-servers/memory.js` (which the in-repo version is already feature-complete and the external repo's data dir is empty). The key work is:

1. Add 5 npm deps to `package.json`
2. Update `setup-mcp.sh` to use `node_modules` paths instead of `npx -y`
3. Eliminate the `paloma-memory` external repo clone
4. Optionally extract the mcp-settings JSON template to a committed file

The `mcp-settings.json` stays outside the repo (API keys). The config generation stays in `setup-mcp.sh`. The bridge loading mechanism (`bridge/config.js`) needs zero changes.

**Biggest risk:** Getting the exact entry point paths right for each npm-installed package. Chart should plan a step to verify these after `npm install`.
