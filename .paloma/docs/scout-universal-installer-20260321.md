# Scout: Universal Installer for Paloma

**Date:** 2026-03-21
**Phase:** Scout (Research)
**Status:** Complete — ready for Chart

---

## Executive Summary

**Recommended approach: Hybrid Shell Script Installer + Managed Node.js Runtime**

After surveying 8 comparable projects and evaluating 10 packaging strategies against Paloma's specific architecture, the recommended approach is a **curl-pipe-bash shell script installer** (modeled after Ollama's installer) that:

1. Downloads and installs a **self-contained Node.js runtime** (via direct binary download) into `~/.paloma/node/`
2. Clones the Paloma repository into `~/.paloma/app/`
3. Runs `npm install` using the managed Node.js
4. Optionally sets up the Python venv for voice TTS
5. Optionally configures systemd/launchd service
6. Provides a `paloma` CLI wrapper that uses the managed Node.js

This approach was chosen because:
- Paloma is a **multi-process Node.js application** with 12+ MCP servers, a WebSocket bridge, and subprocess-spawned AI CLIs — it cannot be meaningfully compiled into a single binary
- The Node.js SEA (Single Executable Application) feature cannot bundle `node_modules/` or spawn child `node` processes for MCP servers
- Docker adds unnecessary complexity for a developer tool that needs to interact with local filesystems, CLI tools, and audio hardware
- A shell script installer is battle-tested (Ollama, nvm, Bun, Deno all use it) and can be audited by security-conscious users

### Process Architecture

Paloma runs as a **process tree**, not a single process:
```
paloma-supervisor.js
├── vite build (one-shot at startup)
├── static-server.js (serves dist/ on :5173)
└── bridge/index.js (WebSocket server on :19191)
    ├── mcp-proxy-server.js (SSE on :19192)
    ├── claude CLI subprocess(es)
    ├── codex CLI subprocess(es)
    ├── copilot CLI subprocess(es)
    ├── gemini CLI subprocess(es)
    └── MCP server subprocesses (12 stdio servers, each a separate node process)
```

This process tree is the **key constraint** — any packaging solution must support spawning child `node` processes.

### External Dependencies (User Must Install Separately)

- **Claude CLI** (`npm install -g @anthropic-ai/claude-code`) — primary AI backend
- **Codex CLI** (`npm install -g @openai/codex`) — optional backend
- **Copilot CLI** (`gh copilot`) — optional backend, needs GitHub CLI
- **Gemini CLI** (`npm install -g @google/gemini-cli`) — optional backend
- **Ollama** (`curl -fsSL https://ollama.com/install.sh | sh`) — optional backend
- **GitHub CLI** (`gh`) — for Copilot backend auth

## Survey of Existing Installer Approaches

### 1. Ollama — The Gold Standard for `curl | bash`

**How it works:**
- Single shell script at `https://ollama.com/install.sh`
- Detects OS (`uname -s`) and arch (`uname -m`)
- macOS: Downloads `.zip`, extracts to `/Applications/Ollama.app`, symlinks to `/usr/local/bin/ollama`
- Linux: Downloads `.tar.zst` (with `.tgz` fallback), extracts to `/usr/local/`
- Creates `ollama` system user, configures systemd service
- Detects GPU (NVIDIA/AMD), installs CUDA drivers if needed
- WSL2 detection and special handling

**Key techniques worth adopting:**
- `main()` function wrapper prevents partial-download execution
- Temp dir with cleanup trap
- `require()` helper checks for needed tools (curl, awk, grep, etc.)
- Graceful sudo escalation (checks if root, falls back to sudo)
- systemd service creation with proper `After=network-online.target`
- GPU detection is optional — core install succeeds without it
- Idempotent — re-running updates the existing installation

**Relevance to Paloma:** High. Ollama is a server process (like Paloma's bridge) that needs to bind ports and run as a service. Their systemd integration is the model to follow.

### 2. nvm — Git Clone + Shell Profile Injection

**How it works:**
- Prefers `git clone` into `~/.nvm/` (shallow clone, depth=1)
- Falls back to downloading individual scripts via curl/wget
- Injects `source` lines into detected shell profile (`.bashrc`, `.zshrc`, `.bash_profile`, `.zprofile`)
- Shell function-based — `nvm` is not a binary, it's a bash function

**Key techniques worth adopting:**
- Shell profile detection (`nvm_detect_profile`) handles every common shell config file
- Idempotent profile injection (checks with `grep` before appending)
- Git-based updates (just `git fetch && git checkout`)
- Warns about existing global npm modules that might conflict

**Relevance to Paloma:** Medium. The profile detection and git-based update mechanism are valuable. The shell function approach doesn't apply since Paloma needs a running server.

### 3. Bun — Download Binary + PATH Setup

**How it works:**
- Downloads pre-built binary from GitHub releases
- Detects OS/arch via `uname -ms`, maps to target triple
- Special handling: Rosetta 2 detection, Alpine/musl detection, AVX2 CPU capability check
- Installs to `~/.bun/bin/`
- Detects shell, appends PATH export to appropriate rc file
- Installs shell completions

**Key techniques worth adopting:**
- CPU capability detection (AVX2) for performance-optimized builds
- Rosetta 2 detection on Apple Silicon Macs
- musl vs glibc detection for Alpine Linux
- Clean `BUN_INSTALL` env var for custom install location

**Relevance to Paloma:** Medium-low. Bun is a single binary — simpler than Paloma's multi-process architecture. But the OS/arch detection patterns are reusable.

### 4. Deno — Minimal Binary Download

**How it works:**
- ~80 lines of shell script (extremely simple)
- Downloads binary from `dl.deno.land`, unzips, makes executable
- Runs Deno itself for shell setup (`deno run jsr:@deno/installer-shell-setup/bundled`)
- Interactive prompts for PATH setup unless `-y` passed

**Key techniques worth adopting:**
- Using the installed tool itself for complex setup (Deno runs a Deno script for shell config)
- The `-y` / `--yes` flag for non-interactive CI usage

**Relevance to Paloma:** Low for the binary approach, but the "use the tool to set itself up" pattern is clever — we could use Node.js to run a setup script after downloading it.

### 5. n8n — npm Global Install + Docker

**How n8n is installed:**
- Primary: `npx n8n` or `npm install -g n8n` (requires Node.js pre-installed)
- Docker: `docker run -it --rm --name n8n -p 5678:5678 n8nio/n8n`
- No custom installer script — relies on Node.js ecosystem or Docker

**Relevance to Paloma:** Shows that a Node.js automation platform can succeed with "just use npm" but they also prioritize Docker for production. Their approach works because they have a simpler architecture (single process, no CLI subprocess spawning).

### 6. Home Assistant — Python Venv Bootstrapping

**How it installs:**
- Multiple methods: Docker, supervised install, Home Assistant OS (dedicated Linux image)
- Python venv approach for "core" install: requires Python 3.12+, creates venv, pip install
- Heavy use of Docker for the full experience

**Relevance to Paloma:** The Python venv bootstrapping pattern is directly relevant for Kokoro TTS. Their multi-tier installation (simple vs. full) is a model for making voice TTS optional.

## Evaluation of Packaging Strategies

### Evaluation Criteria

| Criteria | Weight | Description |
|----------|--------|-------------|
| Zero prereqs | High | User needs nothing pre-installed |
| Cross-platform | High | Linux, macOS, WSL2 |
| Spawns child processes | Critical | Must support Paloma's process tree |
| Update mechanism | High | Seamless updates without losing config |
| Uninstall story | Medium | Clean removal |
| Install size | Medium | Download + disk footprint |
| Build/maintain complexity | Medium | How hard is it for us to maintain |
| Python venv support | Medium | Kokoro TTS voice feature |
| Service management | Medium | systemd/launchd optional setup |

### Strategy Evaluation Matrix

#### A. Shell Script Installer (RECOMMENDED)

**Approach:** `curl -fsSL https://install.paloma.dev | bash` downloads a shell script that installs Node.js (via direct binary download), clones the repo, runs npm install, and optionally configures a service.

| Criteria | Rating | Notes |
|----------|--------|-------|
| Zero prereqs | 9/10 | Needs curl + git (universally available). Installs own Node.js |
| Cross-platform | 10/10 | Shell script works everywhere — Linux, macOS, WSL2 |
| Child processes | 10/10 | Full Node.js runtime, no restrictions |
| Update mechanism | 9/10 | `paloma update` = git pull + npm install |
| Uninstall | 9/10 | `paloma uninstall` = rm -rf ~/.paloma + remove service |
| Install size | 7/10 | ~500 MB (Node.js ~50 MB + node_modules ~424 MB + source ~15 MB) |
| Build complexity | 9/10 | Shell script, no build pipeline needed |
| Python venv | 9/10 | Existing setup-mcp.sh already handles this |
| Service mgmt | 9/10 | systemd/launchd templates easy to generate |
| **Overall** | **9.1/10** | |

**Pros:**
- Proven pattern (Ollama, nvm, Bun all do this)
- Auditable — users can read the script before running
- No build pipeline or CI/CD needed for packaging
- Leverages existing `scripts/setup-mcp.sh` and `scripts/first-run.sh`
- Git-based updates are atomic and reliable
- Node.js isolated in `~/.paloma/node/` — no conflict with user's system Node

**Cons:**
- ~500 MB total install (node_modules is large)
- Requires internet connection for initial install
- npm install can be slow on first run

#### B. Node.js SEA (Single Executable Application)

**Approach:** Compile Paloma's bridge into a single binary using Node.js 20+ SEA feature.

| Criteria | Rating | Notes |
|----------|--------|-------|
| Zero prereqs | 8/10 | Single binary download |
| Cross-platform | 7/10 | Need to build per platform (CI matrix) |
| Child processes | 2/10 | **FATAL: Cannot spawn `node` for MCP servers** |
| Update mechanism | 6/10 | Download new binary |
| Uninstall | 10/10 | Delete the binary |
| Install size | 6/10 | ~100-150 MB estimated per binary |
| Build complexity | 3/10 | SEA is experimental, complex build pipeline |
| Python venv | 1/10 | Cannot bundle Python |
| Service mgmt | 7/10 | Binary can be run as service |
| **Overall** | **3.5/10** | |

**FATAL FLAW:** Node.js SEA bundles ONE script into the Node.js binary. It cannot:
- Include `node_modules/` (must bundle with esbuild/webpack first)
- Spawn child `node` processes (there is no `node` binary — it IS the binary)
- Run MCP servers as stdio subprocesses (they need `node mcp-servers/foo.js`)

Paloma spawns 12+ child Node.js processes. SEA makes this impossible without also installing a separate Node.js runtime, which defeats the purpose.

#### C. Docker / Docker Compose

**Approach:** Containerize Paloma with all dependencies.

| Criteria | Rating | Notes |
|----------|--------|-------|
| Zero prereqs | 5/10 | Needs Docker installed (non-trivial on some systems) |
| Cross-platform | 8/10 | Docker runs everywhere, but macOS perf penalty |
| Child processes | 10/10 | Full OS, no restrictions |
| Update mechanism | 8/10 | `docker pull` new image |
| Uninstall | 9/10 | `docker rm` + `docker rmi` |
| Install size | 6/10 | ~1.5 GB image (Node.js + Python + all deps) |
| Build complexity | 7/10 | Dockerfile + CI for multi-arch builds |
| Python venv | 10/10 | Bundled in image |
| Service mgmt | 7/10 | Docker restart policies, but extra layer |
| **Overall** | **5.5/10** | |

**Pros:**
- Reproducible — identical everywhere
- Bundles Python/Kokoro TTS trivially
- No conflict with host system

**Cons:**
- **Docker is a prerequisite** — violates the "zero prereqs" goal
- **Cannot access Claude CLI, Codex CLI, etc.** on host system (they run outside Docker)
- **Cannot access `~/.claude/`, `~/.config/github-copilot/`** auth credentials
- **Audio passthrough** for Kokoro TTS is extremely complicated in Docker
- **Filesystem access** to user's projects is via mounts (awkward)
- **WebSocket port binding** needs `-p 19191:19191` (easy but another step)
- macOS Docker has ~30% I/O perf penalty due to Linux VM

**Verdict:** Docker works for server-side apps, but Paloma is a **local developer tool** that needs deep host integration (CLIs, auth files, audio, filesystem). Docker creates more problems than it solves.

## Technical Challenges and Solutions

### 1. Bundling Node.js Without Conflicting with User's Existing Node

**Problem:** Many Paloma users will have Node.js installed (they're developers). Installing a system-wide Node.js could conflict.

**Solution: Isolated Node.js in `~/.paloma/node/`**

```bash
# Download Node.js binary directly (no nvm/fnm needed)
NODE_VERSION="22.12.0"  # LTS
ARCH=$(uname -m)
OS=$(uname -s | tr '[:upper:]' '[:lower:]')

case "$ARCH" in
  x86_64)  ARCH="x64" ;;
  aarch64|arm64) ARCH="arm64" ;;
esac

NODE_URL="https://nodejs.org/dist/v${NODE_VERSION}/node-v${NODE_VERSION}-${OS}-${ARCH}.tar.xz"
curl -fsSL "$NODE_URL" | tar xJ -C "$HOME/.paloma/node/" --strip-components=1
```

The `paloma` wrapper script uses this Node.js exclusively:
```bash
#!/bin/bash
export PATH="$HOME/.paloma/node/bin:$PATH"
exec node "$HOME/.paloma/app/scripts/paloma-supervisor.js" "$@"
```

This approach:
- Never touches the system Node.js
- Works even if user has no Node.js
- Specific version we've tested against
- Similar to how fnm/nvm work internally

### 2. Python Venv for Kokoro TTS

**Problem:** Voice TTS requires Python 3.10+ with `kokoro`, `sounddevice`, `markdown`, `spacy` packages. The venv is 7.4 GB.

**Solution: Optional, deferred installation**

The core installer skips Python/voice entirely. Voice is installed on first use or via `paloma setup voice`:

```bash
paloma setup voice
# Detects Python 3.10+, creates venv in ~/.paloma/kokoro_env/
# Downloads models on first speak
```

If Python isn't available, the installer prints a friendly message:
```
Voice TTS (Kokoro) requires Python 3.10+.
Install Python and run: paloma setup voice
Paloma works perfectly without voice — this is optional.
```

**Alternative for future:** Consider `uv` (Astral's Python package manager) which can install Python itself:
```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
uv python install 3.12
uv venv ~/.paloma/kokoro_env --python 3.12
uv pip install kokoro sounddevice markdown spacy
```

This would make Python truly zero-prereq but adds `uv` as an intermediate dependency.

### 3. MCP Servers from npm

**Problem:** 5 MCP servers come from npm packages (`@modelcontextprotocol/server-brave-search`, `server-filesystem`, `@mseep/git-mcp-server`, `@kevinwatt/shell-mcp`, `@thelord/mcp-cloudflare`). They live in `node_modules/` and are spawned as `node node_modules/.../dist/index.js`.

**Solution:** These are installed by `npm install` as part of the normal dependency tree. The installer just needs to:
1. Install the managed Node.js
2. Run `npm install` in the app directory
3. The `mcp-settings.json` generation (existing `setup-mcp.sh`) handles the paths

No special handling needed — this already works.

### 4. External CLI Backends

**Problem:** Claude CLI, Codex CLI, Copilot CLI, Gemini CLI are separate tools the user installs.

**Solution: Detection + guided setup, not bundling**

The installer detects what's available and provides setup guidance:

```
Detected backends:
  ✔ Claude CLI v4.2.1
  ✔ GitHub CLI v2.65.0 (Copilot backend)
  ✗ Codex CLI — install: npm install -g @openai/codex
  ✗ Gemini CLI — install: npm install -g @google/gemini-cli
  ✗ Ollama — install: curl -fsSL https://ollama.com/install.sh | sh

Paloma requires at least one AI backend. Claude CLI is recommended.
```

A future `paloma doctor` command could re-run this detection.

### 5. Port Binding and Firewall

**Problem:** Paloma binds ports 19191 (WebSocket) and 19192 (MCP proxy). Firewall rules may block these.

**Solution:** Both ports bind to `127.0.0.1` (localhost only) — no firewall rules needed. The installer should verify ports are available:

```bash
if lsof -i :19191 >/dev/null 2>&1; then
  echo "Port 19191 is in use. Stop the existing process or configure a different port."
fi
```

### 6. Config Preservation During Updates

**Problem:** `~/.paloma/mcp-settings.json` contains user API keys. Updates must not overwrite them.

**Solution:** Already handled by existing `setup-mcp.sh` — it extracts existing keys before regenerating the file. The update process should be:

```bash
paloma update
# 1. git pull (or download new release)
# 2. npm install (update deps)
# 3. bash scripts/setup-mcp.sh (preserves API keys)
# 4. vite build (rebuild frontend)
# 5. Restart bridge if running
```

### 7. Service Management Across Platforms

**systemd (Linux):**
```ini
[Unit]
Description=Paloma AI Development Partner
After=network-online.target

[Service]
Type=simple
User=%i
ExecStart=/home/%i/.paloma/bin/paloma start --no-browser
Restart=on-failure
RestartSec=5
Environment="HOME=/home/%i"

[Install]
WantedBy=default.target
```

**launchd (macOS):**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>dev.paloma.bridge</string>
  <key>ProgramArguments</key>
  <array>
    <string>/Users/USERNAME/.paloma/bin/paloma</string>
    <string>start</string>
    <string>--no-browser</string>
  </array>
  <key>RunAtLoad</key>
  <true/>
  <key>KeepAlive</key>
  <true/>
</dict>
</plist>
```

**WSL2:** Use systemd if enabled (`/etc/wsl.conf` with `[boot] systemd=true`), otherwise generate a shell alias for manual start. WSL2 systemd support is standard since Windows 11 22H2.

## Recommended Architecture

### Directory Layout

```
~/.paloma/
├── bin/
│   └── paloma              # CLI wrapper script (symlinked to /usr/local/bin)
├── node/                   # Isolated Node.js runtime
│   ├── bin/
│   │   ├── node
│   │   └── npm
│   └── ...
├── app/                    # Paloma source code (git clone)
│   ├── bridge/
│   ├── src/
│   ├── mcp-servers/
│   ├── scripts/
│   ├── package.json
│   └── ...
├── kokoro_env/             # Python venv for voice (optional)
├── mcp-settings.json       # MCP server configuration (user API keys)
├── mcp.json                # Permission/auto-execute rules
├── memory/                 # Semantic memory storage
├── version                 # Installed version tag
└── .paloma-install.json    # Installer metadata (version, date, method)
```

### The `paloma` CLI Wrapper

```bash
#!/usr/bin/env bash
# ~/.paloma/bin/paloma — Paloma CLI wrapper
set -euo pipefail

PALOMA_HOME="${PALOMA_HOME:-$HOME/.paloma}"
PALOMA_APP="$PALOMA_HOME/app"
PALOMA_NODE="$PALOMA_HOME/node/bin/node"
PALOMA_NPM="$PALOMA_HOME/node/bin/npm"

export PATH="$PALOMA_HOME/node/bin:$PATH"

case "${1:-start}" in
  start)
    exec "$PALOMA_NODE" "$PALOMA_APP/scripts/paloma-supervisor.js" "${@:2}"
    ;;
  stop)
    PID_FILE="/tmp/paloma-bridge.pid"
    [ -f "$PID_FILE" ] && kill "$(cat "$PID_FILE")" 2>/dev/null
    ;;
  update)
    cd "$PALOMA_APP"
    git pull origin main
    "$PALOMA_NPM" install
    bash "$PALOMA_APP/scripts/setup-mcp.sh"
    echo "Updated. Restart with: paloma restart"
    ;;
  restart)
    "$0" stop
    sleep 1
    "$0" start "${@:2}"
    ;;
  doctor)
    "$PALOMA_NODE" "$PALOMA_APP/scripts/paloma-doctor.js"
    ;;
  setup)
    case "${2:-}" in
      voice) bash "$PALOMA_APP/scripts/setup-voice.sh" ;;
      *) bash "$PALOMA_APP/scripts/setup-mcp.sh" ;;
    esac
    ;;
  uninstall)
    echo "This will remove Paloma and all its data."
    read -p "Continue? [y/N] " confirm
    [ "$confirm" = "y" ] || exit 0
    "$0" stop 2>/dev/null || true
    # Remove service
    systemctl --user disable paloma 2>/dev/null || true
    rm -f ~/.config/systemd/user/paloma.service 2>/dev/null || true
    launchctl unload ~/Library/LaunchAgents/dev.paloma.bridge.plist 2>/dev/null || true
    rm -f ~/Library/LaunchAgents/dev.paloma.bridge.plist 2>/dev/null || true
    # Remove symlink
    sudo rm -f /usr/local/bin/paloma 2>/dev/null || true
    # Remove installation
    rm -rf "$PALOMA_HOME"
    echo "Paloma has been uninstalled."
    ;;
  version)
    cat "$PALOMA_HOME/version" 2>/dev/null || echo "unknown"
    ;;
  *)
    echo "Usage: paloma {start|stop|restart|update|doctor|setup|uninstall|version}"
    ;;
esac
```

### Phase 1: Shell Script Installer (MVP)
- Install script hosted at `install.paloma.dev` (or raw GitHub URL initially)

- Detect and report AI backend availability

## Open Questions for Chart

1. **Git clone vs. release tarball?** Git clone enables `git pull` updates but requires git. Release tarballs are smaller and don't need git but require a separate update mechanism. Recommendation: Git clone for now, add tarball releases in Phase 3.

2. **Node.js version policy?** Pin to a specific LTS version (e.g., 22.12.0) or track latest LTS? Pinning is safer, but requires installer updates. Recommendation: Pin to specific LTS, update in installer when tested.

3. **Repository URL?** Currently `github.com/adam/paloma` (private?). For public installer, needs to be accessible. Options: public GitHub, self-hosted Gitea, or tarball download.

4. **Service default?** Should the installer set up a systemd/launchd service by default, or only on explicit request? Recommendation: Ask during install (default: no for interactive, yes for `--service` flag).

5. **Multi-user support?** Current design installs per-user in `~/.paloma/`. Should we support system-wide installation? Recommendation: Per-user only for now — simpler, no sudo needed for core install.

6. **Telemetry/analytics?** Should the installer report install success/failure for debugging? Recommendation: No telemetry without explicit opt-in. Privacy first.

7. **Version scheme?** Paloma currently has `"version": "0.0.0"` in package.json. Need a versioning strategy for releases. Recommendation: Semver, start at 0.1.0, use git tags.

8. **Existing `first-run.sh`?** The existing script (`scripts/first-run.sh`) assumes user already has Node.js/npm/git. Should the installer replace it, wrap it, or coexist? Recommendation: Installer replaces first-run.sh for new users; first-run.sh remains for developers who clone manually.
